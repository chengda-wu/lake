# SLO — 服务等级目标

给用户的可量化质量承诺。SLO 是架构设计的**硬约束**：倒逼 Prefill/Decode 物理分离、KV 传输、分层缓存等设计必须在预算内完成。

> 目标值为**初版草案**，待 P7 性能建模校准。标 `[draft]` 者需实测确认可行性。

---

## 延迟

| 指标 | 定义 | 目标（draft） | 约束的设计点 |
|------|------|---------------|--------------|
| TTFT | 请求发出 → 首个 token 返回 | P50 < 400ms, P99 < 800ms | 前缀命中率、prefill 计算量、KV 传输、**执行模式选择** |
| TTFT (D-direct 路径) | APC 命中直跳的 TTFT | P50 < 150ms | APC 命中率、D 节点本地性 |
| ITL | 逐 token 间隔 | P50 < 50ms, P99 < 200ms | decode batch 调度、KV 本地性、投机解码 |
| 队列等待 | 入队 → 开始执行 | P99 < 300ms | 各池容量、弹性扩容时延 |

> TTFT 目标按**主流路径**（PD 分离/混部）设；D-direct 是 APC 命中时的快路径，单独给更紧目标。模式选择本身的开销须 < 5ms（否则吃掉 D-direct 收益）。

## 吞吐

| 指标 | 定义 | 目标（draft） |
|------|------|---------------|
| 单卡 decode 吞吐 | tokens/s/GPU | > 2000 |
| 集群聚合吞吐 | tokens/s | 随节点线性扩展（效率 > 80%） |
| Prefill 吞吐 | tokens/s/GPU | > 5000 |

## 弹性与恢复

| 指标 | 定义 | 目标（draft） | 约束的设计点 |
|------|------|---------------|--------------|
| 扩容生效 | 扩容决策 → Ready 接受请求 | < 10s | 权重预加载、layer-async serve、KV prefetch |
| 缩容恢复 | 节点 Drain → KV 完全写回 Pool | < 5s | 增量写回频率 |
| 故障续推 | Decode 节点崩溃 → 新节点续推首 token | < 2s | KV Pool 命中、拉取带宽 |
| 缩容无丢失 | 缩容后请求成功率 | 100% | KV 已持久化到 Pool |

## 资源效率

| 指标 | 定义 | 目标（draft） |
|------|------|---------------|
| KV 命中率 | 复用 block / 请求总 block | > 60%（公共前缀场景） |
| APC 命中率 | D 节点本地命中前缀 / 可复用前缀 | > 40%（重复请求场景） |
| HBM 利用率 | 有效 KV / HBM 容量 | > 70% |
| 前缀复用率 | 复用 token / prompt token | 场景相关，需建模 |

## 可用性

| 指标 | 目标 |
|------|------|
| 请求成功率（非过载） | 99.9% |
| 数据面单节点故障影响 | 0 个请求失败（自动续推） |

---

## SLO 预算与降级

- 每项 SLO 设**错误预算**（如可用性 99.9% → 月预算 ~43min）。
- 预算耗尽 → 触发降级：降低 batch size 保 ITL、限制并发保 TTFT、拒绝低优先级请求。
- SLO 违反时优先保 Must 特性的核心路径（F1-F5），Should 特性（F6-F8）可先关停让路。

## 校准计划

P7 性能建模阶段，用 micro-benchmark 与端到端实测回填真实数值，修正本表。当前数值仅作 P1 架构设计的初始约束。
