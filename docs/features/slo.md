# SLO — 服务等级目标

给用户的可量化质量承诺。SLO 是架构硬约束:倒逼 Prefill/Decode 分离、KV 传输、分层缓存等设计在预算内完成。

> 目标值为初版草案,待 P7 校准。

---

## 延迟

| 指标 | 定义 | 目标（draft） | 约束的设计点 |
|------|------|---------------|--------------|
| TTFT | 请求发出 → 首个 token 返回 | P50 < 400ms, P99 < 800ms | 前缀命中率、prefill 计算量、KV 传输、**执行模式选择** |
| TTFT (D-direct 路径) | 本地命中直跳的 TTFT | P50 < 150ms | 本地命中率、存储池 HBM 放置本地性 |
| ITL | 逐 token 间隔 | P50 < 50ms, P99 < 200ms | decode batch 调度、KV 本地性、投机解码 |
| 队列等待 | 入队 → 开始执行 | P99 < 300ms | 各池容量、弹性扩容时延 |

> TTFT 按主流路径(PD 分离/混部)设;D-direct 是本地命中(前缀 KV 已在执行节点 HBM)的快路径,给更紧目标。模式选择开销须 < 5ms,否则吃掉本地命中省传输的收益。

## 弹性与恢复

| 指标 | 定义 | 目标（draft） | 约束的设计点 |
|------|------|---------------|--------------|
| 扩容生效 | 扩容决策 → Ready 接受请求 | < 10s | 权重预加载、layer-async serve、KV prefetch |
| 缩容恢复 | 节点 Drain → KV 完全写回 Pool | < 5s | 增量写回频率 |
| 故障续推 | Decode 节点崩溃 → 新节点续推首 token | < 2s | KV Pool 命中、拉取带宽 |
| 缩容无丢失 | 缩容后请求成功率 | 100% | KV 已持久化到 Pool |

## SLO 预算与过载控制

- 每项 SLO 设**错误预算**（如可用性 99.9% → 月预算 ~43min）。
- **过载控制（限并发 / 拒请求 / 按优先级丢弃）归 gateway / 外部控制面**，不在推理系统内执行。推理系统只管执行：忠实执行被分配的请求，对过载的响应是"上报队列与资源指标"供 gateway 决策，而非自行降 batch / 拒请求。
- 推理系统侧的职责边界：Worker 不得为保 SLO 自行降低 batch size 或丢弃请求；队列与 in-flight 指标实时上报 gateway，由 gateway 做入口准入与过载 shedding（限并发/拒请求/按优先级丢弃），已准入请求的 SLO 路由与优先级排队由 Router 负责（不丢请求）。
- 可用性 SLO（99.9%）针对**非过载、已准入**的请求；过载导致的拒绝不计入推理系统的失败率，由 gateway 的过载策略负责。
