# 01 — 目标与非目标

> **设计主线**：本系统**以 KV 为中心**设计——请求是短暂的、节点是可换的，KV 是贯穿多请求、多节点、多轮次的唯一长存物。**agent 多轮对话**是首要驱动场景：其"长共享前缀 + 逐步增长"结构天然契合 KV 池化、前缀复用与反向回传增强（见 [`../architecture/execution-modes.md`](../architecture/execution-modes.md)）。

## 目标

1. **彻底存算分离**
   计算节点不持久化任何推理相关状态（权重、KV cache、调度队列），所有状态归属存储层。算力节点崩溃 / 缩容后不丢失可恢复的推理进度。

2. **KV cache 作为一等公民的分布式资源**
   KV cache 不属于产生它的 GPU，而是属于一个全局池。前缀可跨请求、跨节点、跨用户复用；可按需迁移、驱逐、预取。多轮场景下 decode 延伸的 KV 回流池中，前缀持续生长，下一轮自动命中更长。

3. **Prefill / Decode 物理隔离**
   Prefill（计算密集、长序列）与 Decode（访存密集、逐 token）运行在不同算力池，各自按负载特征独立伸缩，互不阻塞。

4. **秒级弹性**
   算力池可按指标（队列长度、QPS、KV 命中率）秒级扩缩。冷启动（权重加载、CUDA 初始化）需被压缩到可接受范围。

5. **单一事实来源**
   对象存储（S3 类）作为权重、checkpoint、KV cache 持久副本的唯一权威来源，其上叠加分层缓存（GPU HBM → 主机内存 → 本地 NVMe → 远端内存池 → 对象存储）。

## 非目标（至少现阶段）

- **不做训练**：仅推理（含投机解码、prefill 优化），不涉及梯度与参数更新。
- **不绑定单一硬件**：原型以 NVIDIA GPU 为主，但存储/调度抽象应足够通用，不写死 CUDA 细节。
- **不追求单机极致性能**：先验证架构假设，单 kernel 优化不是本仓库重点。
- **不做完整的前端 serving 框架**：不实现 OpenAI 兼容 API 之外的完整产品化能力（鉴权、计费、多租户隔离等）。

## 衡量指标

| 维度 | 指标 |
|------|------|
| 延迟 | TTFT (Time To First Token)、P50/P99 ITL (Inter-Token Latency) |
| 吞吐 | tokens/s/GPU、aggregate tokens/s |
| 弹性 | 扩容生效时延、缩容后 KV 恢复时延、节点崩溃后请求重路由时延 |
| 资源效率 | KV cache 命中率、HBM 利用率、前缀复用率 |
| 成本 | 每百万 token 成本、闲置算力占比 |
