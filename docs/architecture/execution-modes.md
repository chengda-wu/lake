# 执行模式与 KV 流转

把 [`../features/features.md`](../features/features.md) "执行模式"节细化到架构层:一个请求的 KV 从哪来、写到哪、谁调度、失败怎么办。

本文以 **KV 为中心**展开。请求短暂、节点可换,KV 是贯穿多请求、多节点、多轮次的唯一长存物;三种执行模式本质都是 KV 的流转与生长。

## 设计主线:KV 生命周期

架构以 KV 为中心,**agent 多轮对话**是首要驱动场景。agent 结构天然是"长共享前缀 + 逐步增长":系统提示/工具描述/few-shot 超长且不变,历史轮逐轮 append,前缀单调增长。

一个 KV 前缀的完整生命周期都在存储池:

```
诞生(prefill 产出) → 共享(多请求复用) → 生长(D 回传延伸) → 老化(冷热下沉) → 终结(GC)
```

agent 多轮把这条主线显式化:每轮增长的 KV 写回存储池 → radix 长一截 → 下一轮自动命中更长前缀。这是反向回传机制(见下)的驱动场景。

## 存储池视角:不区分 P/D

从存储池视角,执行节点都是同质的计算服务——prefill、decode、draft、encoder 或任何未来角色。存储池只关心**这次请求要不要跨节点搬 KV**,不关心节点叫什么。

P/D 是调度层对节点的角色分配,不进存储池时序。本文时序用"执行节点 A/B"中性称呼;谁当 P、谁当 D 由调度器定(见 [`scheduling.md`](scheduling.md))。

## 两条时序

| 时序 | 含义 | 对应原模式 |
|------|------|-----------|
| **本地完成** | 请求在一个节点从头跑到尾,KV 全程本机 L0,无跨节点传输 | 混部 / D-direct |
| **跨节点传输** | 请求生命周期内某段 KV 要从一个节点搬到另一个 | PD 分离、换节点、反向回传 |

### 时序一:本地完成

请求在单节点完成 prefill + decode,KV 全程本机 L0。存储池只做:查视图、接收 prefill 产出写回、供 decode 读。

```
请求泳道(5ms 预算内):
  Router → 存储池一次查询(前缀解析 + 位置)
        ← 可复用 block 列表 + 各自位置(含"是否已在某 HBM"= 本地命中判定)
       → 路由到目标节点(本地命中节点优先)
       → 本机 prefill(残差或完整,视本地命中)→ 本机 L0
       → 本机 decode → 增量 KV 异步写回存储池
```

入口由本地命中与否区分 prefill 工作量(调度层细节,非存储池时序分支):

- **本地命中**(前缀 KV 已被存储池预放置到本机 HBM):残差 prefill + decode,零传输。对应 D-direct。
- **本地不命中**(前缀 KV 仅在 Pool / 未命中):完整或增量 prefill + decode,本机完成。对应混部。

两者共享同一条单节点时序。本地完成不等于绕过存储池——本机 L0 仍是存储池统一管理的载体,prefill 产出仍写存储池、decode 仍从存储池读,省的是跨节点传输。

> **预留(残差 prefill 重算范围)**:当前残差 prefill 定义为"对未匹配尾部做增量 prefill"。r-type SWA 尾段重算优化(见 [`compute-layer.md`](compute-layer.md) "r-type SWA 前缀复用的尾段重算优化")要求重算一段**已匹配前缀的尾部**以刷新 SWA 窗口(非未匹配段、仅写 SWA 模块)。执行路径需预留区分"增量 prefill(未匹配尾)"与"刷新重算(已匹配尾,仅 SWA 写)"两种重算语义,暂不实现但接口先留。

### 时序二:跨节点传输

请求生命周期内某段 KV 要从一个执行节点搬到另一个,存储池负责搬运。分两个方向:

**正向(产出 → 消费):服务本次请求**

A prefill 产出新 KV → 写存储池 → 存储池把这段 KV 放置到 B 的 HBM → B 消费。对应 PD 分离(P→D)。放置的是新产出 KV,必须现放,属请求路径内,与计算重叠流水化。

```
A prefill → 产出 KV 写存储池(L0 即本机)→ 存储池放置到 B 的 HBM(与 A 计算重叠)
                                                         ↓
                                                      B decode
```

**反向(消费 → 池):增强未来命中**

B 在 decode 中延伸了前缀(生成新 token 的 KV)→ 这段延伸 KV 回传存储池 → radix 生长 → 未来请求(如下一轮 agent)查池时命中更长前缀。

```
B decode 生成延伸 KV → 异步回传存储池(落 L2 + 更新 radix)
                              ↓
                    前缀树生长,下次命中边界前推
```

反向回传不为本次请求服务,是为未来请求的前缀增强攒数据。agent 多轮的核心就在此:每轮增长的 KV 回流,下一轮自动命中更长。

正向流新产出 KV、服务本次;反向流延伸 KV、服务未来,两者不可混作一谈。

**第三条方向 D→P(消费 → prefill,服务下一轮)**:agent 多轮里,上一轮 decode 产出的延伸 KV 是下一轮 prefill 的输入前缀。这条 KV 不必绕一跳存储(先落池再被下轮拉),可直接由 decode 侧喂回 prefill 节点——即 [DualPath](../research/dualpath.md) 的 storage-to-decode 路径原生支持。它与反向回传并行:D→池为所有未来请求攒前缀(radix 生长),D→P 为紧邻的下一轮直接喂(若所需 KV 已在 D 的 HBM 则零存储读取直传)。完整流程与双网络选路见 [`data-flow.md`](data-flow.md) §3.4、[`kv-cache-pool.md`](kv-cache-pool.md) "双网络路径"。

## 前置设计决策

以下已在别处定稿,本文沿用:

- **radix tree 归存储池**,按 `model_id` 分命名空间(见 [`kv-cache-pool.md`](kv-cache-pool.md))。Router 一次查询同时拿到可复用 block 列表与各自位置,前缀复用与本地命中判定一跳完成,守住 5ms 模式选择预算。
- **放置职责边界(方案 Z)**:存储池按热度主动预放置高频前缀 KV 到 HBM 并发布位置视图;调度器读视图组 batch(本地命中优先),不反向指挥放置。单向耦合(见 [`storage-layer.md`](storage-layer.md))。
  - 时序一的本地命中依赖存储池**后台**预放置(请求到达前完成),请求路径只查不放。
  - 时序二正向的新 KV 放置在**请求路径内**(新产出无从预放),与计算重叠。
- **decode 增量写回双重目的**:① 容错(崩溃续推的 checkpoint);② 前缀生长(延伸 KV 回流,radix 增长)。写回频率 N 的策略(多轮 agent vs 单轮)留开放,待 P7 校准。

## 模式选择

模式选择是调度层职责,深决策树见 [`scheduling.md`](scheduling.md) / `data-flow.md`(P1)。根本指引:

- 本地命中 → 时序一,残差 prefill(D-direct)。
- 本地不命中、单节点完成划算(prompt 短 / 传输成本 > 分离收益)→ 时序一,完整/增量 prefill(混部)。
- 跨节点传输划算(prefill 大、各节点有空闲、带宽充裕)→ 时序二正向(PD 分离)。
- decode 延伸了前缀(多轮 agent)→ 时序二反向回传。

阈值待 P7 校准。

## 失败处理

执行失败(节点故障/超时)→ 触发 F4 重新路由 → 调度器依最新状态重跑模式选择(纯函数),不设 mode-to-mode 降级阶梯。失败分支见 `data-flow.md`(P1)。

## 职责边界

| 关注点 | 归属 |
|--------|------|
| KV 放置 / 冷热 / 生命周期 / radix / 位置视图 | 存储池 |
| 节点角色分配(P / D / draft) | 调度层 |
| 模式选择决策树、batch 组成 | 调度层 |
| KV 跨节点传输的执行 | 存储池(数据面) |
| decode 增量写回的触发与频率 | 存储池发起(基于引擎 `publish` 产出);频率 N 留 P7 |

## 开放问题

- decode 写回频率 N:多轮 agent(重前缀增强时效,N 小)vs 单轮(重带宽/容错,N 大),待 P7。
- 反向回传的 radix 增长时效:写回到 radix 可见的滞后上限,影响下一轮命中增强。
- 时序二正向"放置与计算重叠"的流水线深度与 prefill 层数对齐(见 [`kv-cache-pool.md`](kv-cache-pool.md) "分块流水线")。
