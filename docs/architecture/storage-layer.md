# 03 — 存储层

存储层是"彻底存算分离"的根基。它托管两类数据：**权重**与 **KV cache**，并提供分层缓存。

**存储池是长期存续、模型无关的独立基础设施**：同一池同时承载多个 `(model_id, revision)` 的权重与 KV，可对接任意模型；模型上下线与池生命周期解耦。池提供按模型的空间分配/扩缩容、GC、碎片整理能力（详见 [`kv-cache-pool.md`](kv-cache-pool.md) 的"多模型生命周期 / 空间分配与扩缩容 / GC / 碎片整理"节）。

## 数据模型

### 权重 (Weights)
- 不可变，按 `(model_id, revision)` 寻址。
- 粒度：按 layer / tensor 分片存储，支持并行加载与按需加载（lazy layer load）。
- 格式：原始 fp16/bf16，可选量化副本 (int8/int4/fp8)。量化版本作为独立 artifact，不在线反量化。

### KV cache
- 可变、有生命周期。按 `(model_id, layer, block_index, token_range)` 寻址。
- 粒度策略（待定）：
  - **per-block**（如 vLLM 的 block）：复用友好，元数据多。
  - **per-layer**：传输批量好，复用粒度粗。
  - **per-sequence**：最简单，复用差。
  - 倾向 **per-block + 前缀树（prefix tree）** 索引，支持前缀复用。
- 持久化：热数据在 RAM/NVMe，冷数据可落对象存储（长上下文会话、共享系统提示词的 KV）。

## 分层缓存

**关键设计点（彻底存算分离）**：L0–L4 五层**全部由存储池统一管理**（放置、驱逐、副本、元数据）。计算节点不拥有任何内存——HBM/RAM/NVMe 是存储池的物理载体，不是 worker 私有状态。计算服务向存储池申请"把所需 KV 放进本机 HBM"，而非自行管理本地缓存。因此不存在计算层私有的易失缓存，所有 KV 位置（包括哪段 KV 在哪个节点 HBM）均为存储池权威元数据。

| 层级 | 介质 | 容量 | 延迟 | 驱逐策略 | 管理主体 |
|------|------|------|------|----------|----------|
| L0 | GPU HBM | 极小 | ~ns | 存储池放置/驱逐 | 存储池统一（元数据强一致，物理载体在计算节点） |
| L1 | 主机 RAM (per-node) | 小 | ~μs | LRU | 存储池统一 |
| L2 | 本地 NVMe (per-node) | 中 | ~10μs | LRU + TTL | 存储池统一 |
| L3 | 远端内存池 (RDMA) | 大 | ~10-100μs | 全局 LRU | 存储池统一（强一致元数据） |
| L4 | 对象存储 (SSOT) | 无限 | ~ms | 永久（带版本） | 唯一权威 |

> **物理约束**：L0–L2 统一管理是**元数据/逻辑层面**的统一（谁在哪个物理介质、何时驱逐、副本关系），物理数据访问仍受 GPU 本地性约束——attention kernel 读 KV 必须在本机 HBM，无法高效跨节点直接读他人 HBM。因此同一 batch 各 sequence 的 KV 必须已在同一 GPU HBM（本地命中），否则需先由存储池补拉放置。放置与 batch 的职责边界见下"放置与 batch 的职责边界（方案 Z）"。

**读取路径**：L0 → L1 → L2 → L3 → L4，逐层回填上层。
**写入路径**：Prefill 产出 → L0（产出即属存储池，HBM 为其当前放置层）→ 异步写 L1/L3 → 按热度决定是否落 L4。

## 冷热与生命周期管理

存储池对每个 KV block 全权决定冷热判定、层间放置、提升/下沉、驱逐与生命周期终结——贯穿 L0→L4 全链路，是"彻底存算分离"下存储池统一管理的核心职责。

### 层间副本 vs 移动

- **L0/L1 做副本**：上层是下层的缓存副本（L0 是 L1 的缓存、L1 是 L2/L3 的缓存）。上层丢了下层还在，回填快；代价是费空间（L0/L1 小，可接受）。
- **L2/L3/L4 之间按移动**：同层冗余无意义，省空间。
- **L4 永久权威**：唯一不可丢的副本。L4 缺失才视为 block 不存在。

数据模型影响：一个 block 在元数据中有**多层位置**（L0/L1 各存一份缓存副本，L2/L3/L4 三选一），而非单一位置。block 的 `locations` 是"层→物理位置"的集合；L0/L1 缺失只是缓存未命中，L4 缺失才视为不存在。

### 冷热判据

| 维度 | 性质 | 作用 |
|------|------|------|
| 引用数（in-flight ref） | 硬门槛（布尔） | >0 → 冻结迁移/驱逐，保护在途请求 |
| 访问频次（frequency） | 连续信号 | 热度主信号，驱动 promotion |
| 最近访问（recency） | 连续信号 | 防"曾经热、现在凉"占上层，驱动 demotion |
| 前缀亲和（公共前缀加权） | 修饰项 | 驱逐时 tie-breaker，公共前缀不易驱逐 |

聚合方式（非简单加权和）：
- 引用数 >0 → 冻结，不进冷热排序（正确性约束）。
- 否则 **热度分** = f(访问频次, recency)，驱动 promotion/demotion。
- 驱逐/下沉排序时，公共前缀 block 给予加权保护。

频次采用 **LFU-Aging（衰减）** 而非滑动窗口：block 数量级大，滑动窗口元数据吃不消；衰减状态小、近似够用。权重与衰减率待 P7 校准。

### 迁移触发：主动为主 + 被动兜底

- **主动**：存储池后台扫描，按热度分 promotion（热块往上提、L0 按热度预放置）/ demotion（冷块往下沉）。贯穿全层，保持上层常驻新鲜热数据、热块预提减少首次 miss。
- **被动**：读 miss 回填、写满驱逐，作为即时响应兜底。

主动迁移是主路径——否则下层冷块淤积、热块首次必 miss，本地命中率起不来。

### 带宽预算

迁移 / GC / 碎片整理**共享一个"后台任务带宽池"**，内部按优先级调度（迁移/GC/碎片整理动态优先级）。总预算仍 < 10%（不额外放宽，见 [`../features/slo.md`](../features/slo.md)）。

### 放置与 batch 的职责边界（方案 Z）

存储池的 HBM 放置与计算层 batch 组成**单向耦合**：

- **存储池主动放置**：按热度/前缀亲和主动把高频 KV 预放置到某些节点 HBM，并发布"哪些 KV 在哪个 HBM"的位置视图。存储池**不知道**任何 batch 意图，只做通用热度放置。
- **调度器读视图组 batch**：组 batch 时读存储池位置视图，优先把已在同一 HBM 的 sequence 组进同一 batch（本地命中 → D-direct），缺失的再让存储池补拉。调度器不反向指挥存储池放置。

信息流单向（存储池发布视图 → 调度器消费），职责清晰。代价：存储池按热度做通用预判，未必对齐每个具体 batch——某冷门前缀被组进 batch 时无本地命中，需临时补拉。

## KV Pool 架构（详见 05）

存储层的 L3（远端内存池）是 KV Pool 的物理载体，由一组 KV Node 组成：
- 每个 KV Node 贡献主机 RAM + NVMe。
- 通过 RDMA 暴露 KV block 的读写。
- 元数据（哪些 block 在哪个 KV Node）由存储池控制面维护。

> 注：L0–L2 虽物理载体在计算节点，但其放置/驱逐元数据同样由存储池统一管理；L3 由专职 KV Node 承载。五层共用一套存储池元数据与 API。

## 一致性模型

- 权重：不可变，无一致性问题，靠缓存失效（revision 变更）。
- KV cache：**写一次读多次**。Prefill 写入后不可变（针对该 token range）；后续 token 的 KV 是新 block。因此不需要多写者并发控制，只需单写者屏障。
- 故障恢复：KV block 写入 L3/L4 后才认为 Prefill 完成；崩溃可从最近的 KV checkpoint 续推。
